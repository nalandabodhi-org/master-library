# Code Review: `workers/ingestion`

**Date:** 2026-06-14  
**Branch:** review  
**Scope:** All source files in `workers/ingestion/src/` and `tests/`  
**Test run:** 56 tests pass (excluding `test_entrypoint.py` which errors on import)

---

## Summary

The project has a solid architectural skeleton: clean module separation, a working state machine, good pure-function discipline for URL builders and payload constructors, and meaningful unit tests for the parts that exist. However, **the two most critical jobs (`run_trigger_job`, `run_reconciliation_job`) and the webhook write-back are all stubs** — the system cannot actually ingest anything end-to-end today. Several bugs also exist that would surface immediately once the stubs are filled in. These are detailed below, ordered by severity.

---

## Bugs

### 1. `_dispatch_http` is sync but called with `await` — **runtime crash**

**File:** `src/publication_flow.py` lines 22, 151, 211

`_dispatch_http` is defined as a plain `def`, not `async def`. It is called with `await` on lines 151 and 211. When the non-injected (real Cloudflare Workers) path is taken, this will raise a `TypeError` at runtime.

```python
# current — WRONG
def _dispatch_http(url, options, http_client=None):
    if http_client is not None:
        return http_client.request(url, options)  # async call returned, not awaited
    return fetch(url, options)

# fix
async def _dispatch_http(url, options, http_client=None):
    if http_client is not None:
        return await http_client.request(url, options)
    return await fetch(url, options)
```

### 2. Pagination bug in `read_airtable_records` — **only ever fetches page 1**

**File:** `src/airtable_client.py` lines 186–192

`url` is set to `""` immediately before the separator check, so `"?" in url` is always `False`. The separator is therefore always `"?"`, which would double-up the query string when `query` is non-empty (e.g., `...?filter=foo?offset=abc`). Fix:

```python
# current — WRONG
url = ""
if "offset" in data:
    separator = "&" if "?" in url or "?" in build_airtable_read_url(...) else "?"
    base_url = build_airtable_read_url(base_id, table_name, query)
    url = f"{base_url}{separator}offset={data['offset']}"

# fix — separator is always & because base_url already contains ? when query is set
url = ""
if "offset" in data:
    base_url = build_airtable_read_url(base_id, table_name, query)
    separator = "&" if "?" in base_url else "?"
    url = f"{base_url}{separator}offset={data['offset']}"
```

### 3. Cloudflare Stream API endpoint is wrong — **all imports will fail**

**File:** `src/stream_client.py` line 137

The URL-based import endpoint in the Cloudflare Stream API is `POST /accounts/{id}/stream/copy`, not `/accounts/{id}/video`. The request body for a URL copy also uses `{"url": "...", "meta": {...}}` at the top level, not the `input_type`/`source` structure currently used. This will produce 404s or 400s in production.

```python
# current — WRONG
"url": STREAM_API_BASE.format(account_id=account_id) + "/video"

# fix
"url": STREAM_API_BASE.format(account_id=account_id) + "/stream/copy"
```

The body must also match the copy endpoint's schema — verify against the [Cloudflare Stream docs](https://developers.cloudflare.com/stream/uploading-videos/upload-from-link/) and update `build_stream_import_request` accordingly.

### 4. Webhook HMAC verification is never called — **security gap**

**File:** `src/entry.py`, `src/webhook_handler.py`

`verify_stream_webhook` is implemented in `stream_client.py` and tested, but is never invoked from `entry.py` or `handle_webhook_payload`. Any HTTP POST to `/webhooks/stream` is accepted unconditionally. The signature header (`Cloudflare-Stream-Webhook-Signature`) and `WEBHOOK_SECRET` env var are documented but wired to nothing.

```python
# entry.py fetch handler should do this before dispatching
import json
from src.stream_client import verify_stream_webhook

# inside fetch():
raw_body = await request.bytes()
if not verify_stream_webhook(dict(request.headers), raw_body, env.WEBHOOK_SECRET):
    return Response("Unauthorized", status=401)
payload = json.loads(raw_body)
```

### 5. `reconcile_state` forces a silent intermediate transition — **misleading state**

**File:** `src/reconciliation_job.py` lines 7–11

When `current_status == "ingesting"` and Stream reports `readyToStream`, the code transitions through `"processing"` first (to satisfy the state machine), then to `"ready_to_stream"`. This intermediate state is never written to Airtable; it only exists in memory. This will cause confusion when the job is wired to Airtable writes — the caller receives `"ready_to_stream"` but the record skipped `"processing"` in Airtable. Either permit `ingesting → ready_to_stream` directly in the state machine, or explicitly write the `"processing"` state to Airtable before advancing.

## Incomplete Implementations (Stubs)

### 6. `run_trigger_job` does nothing

**File:** `src/trigger_job.py`

```python
async def run_trigger_job() -> dict:
    return {"scanned": 0}
```

The full flow — scan Airtable for `ready_for_ingest`, validate eligibility, claim the record, build the R2 URL, call `execute_publication_flow` — is spec'd and partially built in `publication_flow.py` and `eligibility.py`, but none of it is wired here. This is the core of the system.

### 7. `run_reconciliation_job` does nothing

**File:** `src/reconciliation_job.py`

```python
async def run_reconciliation_job() -> dict:
    return {"scanned": 0}
```

`reconcile_state` is correctly implemented and tested, but the job that scans Airtable for stuck records and calls Stream's status API is absent.

### 8. `handle_webhook_payload` only interprets — never writes back

**File:** `src/webhook_handler.py`

The handler interprets the webhook and returns the next state, but never:
- Updates Airtable with the new sync status
- Writes `Cloudflare Play URL`, `Cloudflare iframe URL`, `Ready to Stream At` back to the record
- Triggers the downstream handoff when `ready_to_stream`

`build_airtable_playback_update_payload` and `build_handoff_payload` exist but are dead code.

### 9. `env` is never passed to jobs or handlers

**File:** `src/entry.py`

```python
async def scheduled(self, controller, env, ctx):
    if controller.cron == "*/5 * * * *":
        await run_trigger_job()          # env dropped
```

When `run_trigger_job` and `run_reconciliation_job` are implemented, they'll need API tokens from `env`. Both async jobs should accept `env` (or a `Config` object extracted from `env`) as a parameter.

## Design Issues

### 10. `publication_flow.py` duplicates `airtable_client` logic

Lines 197–220 of `publication_flow.py` hand-roll an Airtable PATCH request that is functionally identical to `update_airtable_record` in `airtable_client.py`. In the non-injected path this creates a second maintenance surface. The non-injected path should call `update_airtable_record` directly rather than rebuilding the same request.

### 11. `page_size` parameter in `read_airtable_records` is accepted but not applied

**File:** `src/airtable_client.py` line 145

The `page_size` param is documented and accepted but never appended to the URL query string (`pageSize=N`). Airtable defaults to 100, so this is a no-op now, but it's a misleading API surface.

### 12. `AIRTABLE_FIELDS` is defined but not applied to queries

**File:** `src/airtable_client.py` line 14

The constant lists the fields the worker cares about, but `read_airtable_records` never passes `fields[]` query parameters to Airtable. Airtable returns all fields on every record, increasing payload size unnecessarily. The list should be serialised into the query string (e.g., `fields[]=Sync+Status&fields[]=Slug&...`).

## Test Coverage Gaps

| Gap | Impact |
|-----|--------|
| `test_entrypoint.py` fails to collect due to `_cloudflare_compat_flags` import error | Entrypoint routing tests are silently skipped |
| No tests for async HTTP helpers (`read_airtable_records`, `update_airtable_record`) | Pagination bug and error handling are untested |

| No end-to-end flow test (trigger → publication → webhook → handoff) | Integration behaviour is unverified |

For `test_entrypoint.py`: the `workers` package requires `_cloudflare_compat_flags` which is part of the Workers runtime shim and is not available in a plain pytest environment. The tests in that file need to mock the `workers` import at the module level, or be restructured to avoid importing from `workers` at collection time.

## Minor / Style

- `src/stream_client.py` has `__body_json` defined with a dunder prefix — conventionally this signals name-mangling in classes. At module level it's harmless but confusing; rename to `_body_json`.
- `src/domain/` and `src/lib/` appear to be missing `__init__.py` files (not visible in the tree). Verify these exist; tests pass because pytest adds paths directly, but the Worker runtime may not.
- `json` is imported inside `__body_json` as a "local import to avoid stdlib at top-level" — this is unnecessary caution in Python 3.13 Workers and adds noise.
